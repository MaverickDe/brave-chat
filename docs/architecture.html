forward Architecture

forward What is the point of this architecture?

The reason why Brave Chat's architecture uses so many different technologies is that each one addresses a different type of problem in the best way possible.

forward Why relational databases?

For example, A key feature of relational databases is that they don’t give up on consistency and data integrity. They also use pessimistic locking, which kind of useful when transactions that violate synchronization are blocked to avoid race conditions and conflicts. However, relational databases weren’t designed to be clustered, although possible.

forward Why Redis pub/sub?

I also opted to use Redis for Brave Chat because it's fast and reliable. Redis was utilized to send messages to recipients in a pub/sub fashion so that if the user is unreachable for any reason, the messages will still exist and can be delivered as soon as the user becomes available again. This ensures that no messages are lost and that our users always receive the information they need on time.

forward Why Deta drive?

I chose a Deta drive to save media files. This is because Deta has a free trial and provides easy-to-use APIs. This helps us keep our costs down while still providing a robust chat experience for the platform users.

forward Why SingleStoreDB?

Scalability: The database should be able to handle new data without performance degradation, so I don't have to reconsider the solution.

Stability: The database should be able to handle failures gracefully. Data should be replicated and backed up so that it can be recovered in the event of a failure.

Ease of use: Developers with decent experience using SQL database should be able to use the database without needing to be database experts. The database should have an intuitive interface that is easy to use.

Low maintenance: We need something that is low maintenance and will work once we set it up. This will help us keep our costs down and ensure that our database is always up and running.

SingleStore is fulfilling most of the above requirements. That’s basically why I chose to use SingleStore for the chat application. SingleStoreDB can be clustered both for replication and for sharding, and its distributed architecture is based on a master-aggregators model.
